# M0.2 — capsule arena (pure form, scalar-in/-out) vs RC

*The cost curve left open in T1b, now measured. The capsule arena is
built (no longer a stub): `jrt_arena_push` before the body routes all heap
allocations into a private bump arena (immortal → no RC, no collector),
`jrt_arena_pop` frees them en bloc. ONLY scalar-in/-out (hard errors otherwise):
no object pointer survives the arena → true fault containment without deep copy.*

## What is now GUARANTEED (correctness, the actual promise)
- Body allocations go into the arena, are immortal (retain/release/collector
  no-op), and are freed en bloc. Heap balance: **0 heap-live** (arena not in
  `live_objects`, freed by pop). Verified for single AND multiple capsules.
- No pointer out: an object result = compile error; a bug in the body can by
  construction only hit the arena. This is the isolation/containment guarantee
  from CAPSULE-EVALUATION.md — now implemented, not merely asserted.
- Bugfix along the way: the body's ref locals are set to null BEFORE the pop,
  otherwise the backend frees them at function end (header load after pop =
  use-after-free → segfault with two sequential capsules). Fixed + regression test.

## Perf: capsule arena vs plain RC (build list → scalar sum, -O2, best-of-9)
Both programs identical except for the `capsule` wrapping; both heap-allocate
(2× `New`, 0× `StackNew`), and the result is a scalar.

| N (nodes) | capsule arena | plain RC | Speedup |
|---|---|---|---|
| 800 000 | 0.074 s | 0.033 s | 0.45× |
| 3 200 000 | 0.120 s | 0.131 s | 1.09× |
| 12 800 000 | 0.266 s | 0.495 s | **1.86×** |

**Asymptotically the arena wins ~2×** (arena ~16 ns/node, RC ~38 ns/node —
no retain/release per allocation, O(1)-like teardown instead of an O(n) release
cascade). **But it loses below ~2.5M nodes** because of arena fixed costs (chunk
malloc + per-object zeroing). Crossover ~2.5M.

## Honest assessment (the T1b curve, now with numbers)
- The "lots of work → small result" bet pays off in SPEED only at
  **truly large** allocation volume (>2.5M). Below that, plain RC is
  faster — on this workload RC is cheap (glibc tcache `calloc` with
  lazy zero pages + fast release cascade), so the arena fixed costs eat up the
  teardown advantage.
- **The main value of the arena is NOT speed, but the guarantee:**
  deterministic, RC-/collector-free, isolated execution with fault containment
  for risky processing with a small output. The perf gain is a bonus at
  large N, not the main argument.
- Consistent with the gate result (M0.1c): the arena was never the O(n²) rescue
  (there was none) and, even as a perf feature, is only a win at large N.
  Its reason for existing is isolation, not throughput.

## Open
- Object-in/-out (deep-copy-in/-out with a cycle visited set) — the other half
  of the pure form; until then scalar-only.
- Lower the arena fixed costs (larger chunks, chunk recycling across capsules).
- The workload where the arena wins STRUCTURALLY: topology-mutating graphs where
  region inference CANNOT prove the RC away (M0.1c) — there the arena is the
  lever instead of RC. Not yet measured.
