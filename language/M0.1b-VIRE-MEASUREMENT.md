# M0.1b — the risk path, executed on Vire IR for the first time

*Context: The M0 gate (`M0-MEASUREMENT.md`) flagged the RC/cycle-collector path red as
the core risk. Up to this point it **never** ran on Vire-generated objects — all
test objects were stack-allocated (small, acyclic, local graphs), "0 live"
was trivially satisfied and proved nothing. This measurement enters the path for the
first time.*

## Why it never fired before
The escape analysis (inherited from the backend, provably operates on Vire IR:
post-solver IR shows `StackNew`) stack-allocates every graph it proves to be
frame-local and non-escaping — **including cycles**, as long as they are
frame-scoped (an `a<->b` pair in `main` that never escapes needs no
collector: it dies with the frame). To force the heap/RC/collector path, an
object must have a **dynamic, loop-carried lifetime** that the analysis cannot
bound statically.

## Forced heap: loop-carried list (RC, acyclic)
`build_and_sum(n)` builds a list of length `n` (each node points to the
previous one → alias across the loop back-edge → **not** stack-allocatable). The IR
shows `New` (not `StackNew`); the LLVM calls `jrt_alloc`.

Measurement (`-O0` measurement mode, because `-O2 -flto` eliminates dead alloc/release pairs —
correct result, but counters stay 0; the solver runs fully in **both** modes):

| Program | allocated | live at end |
|---|---|---|
| `build_and_sum(1000)` + `build_and_sum(100000)` | 101 000 | **0** |

The RC cascade release (free head → free `next` → …) frees the entire
list. **No stack overflow** at depth 100 000 — the iterative drop
worklist (from the M0 collector fix) also carries on Vire objects.

## Forced cycle: ring (only the collector can reclaim it)
`make_ring(n)` builds a chain and closes it (`prev.next = head`) into a ring.
At the end of the function the **external** refcount is 0, but the internal refs
hold every node at ≥1 → **RC alone leaks** → only the cycle collector can reclaim.
`Node.next: Node` is type-cyclic → not provably acyclic → the collector is
linked in (no `-DFASTLLVM_NO_CYCLES`).

**Reclaim + scaling** (one ring, growing):

| n (ring size) | wall time | live at end |
|---|---|---|
| 20 000 | 4.7 ms | 0 |
| 80 000 | 9.1 ms | 0 |
| 320 000 | 27 ms | 0 |

16× size → ~6× time: **roughly linear**. Up to 320 000 cyclic nodes no
stack overflow, everything reclaimed.

**Repeated mid-run collection** — exactly the M0 killer pattern (the O(n²) blowup
came from the hot loop leaving releases pending, and the collector at threshold
fully scanning the *entire* live set). Many rings allocated+dropped in a loop
fire the collector repeatedly:

| rings × 500 nodes | allocated | wall time | live at end |
|---|---|---|---|
| 200 | 100 000 | 10 ms | 0 |
| 800 | 400 000 | 37 ms | 0 |
| 3 200 | 1 600 000 | 128 ms | 0 |

16× work → ~12× time: **linear, no O(n²) recurrence.** The adaptive-
threshold fix (2×live) from the M0 collector rework lives in the shared
runtime and carries on Vire objects.

## What this proves — and what it explicitly does NOT
**Proven (for the first time, on running Vire code, not in argument):**
- Vire objects go through the same `New`/`GetField`/`PutField` ops and the same
  runtime as the Java path; the RC cascade release balances to **0 live**.
- The cycle collector reclaims cyclic Vire graphs (0 live), including
  repeated mid-run collection, and **scales linearly** up to 1.6 M objects —
  the M0 O(n²) blowup does not reappear.

**NOT proven (honestly open):**
- **No throughput comparison against Rust.** The M0 number (108×) was a *ratio*
  to Rust indices. Here there is no matched Rust baseline → "X× Rust" is NOT
  established by this measurement, only "balances correctly + scales linearly".
- **Not the full PageRank workload.** What is measured is build-and-drop of
  rings (fires repeated collection — the risky mechanism). The M0
  workload mutates a *shared* graph in place over 40 iterations; that is
  a sharper sharing/mutation test and still open.
- **`-O0` measurement mode** — throughput at `-O2` diverges (there dead graphs
  are eliminated entirely; a throughput match would require an observable sink).
